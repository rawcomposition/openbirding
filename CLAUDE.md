# OpenBirding

## Code comments

NO code comments. Default to zero. Code must be self-documenting through naming and structure. The ONLY exceptions — the only times a comment is permitted — are: (1) upstream bug workarounds (link the issue/PR), (2) non-obvious regex, (3) protocol/spec/hardware quirks that look wrong but are correct, (4) legally required notices. If a comment would explain what the code does, that is a bug — fix the naming, do not write the comment. When in doubt, no comment.
